## 素材 2：Anthropic 教 Claude 當化學家

Anthropic 發了一篇研究文章《Making Claude a Chemist》。不是比喻，是真的讓 Claude 去跑化學實驗流程。

文章提到他們讓 Claude 操作實驗設備、分析分子結構、設計合成路徑。從「文字模型」到「實驗室助手」，這跨度比想像中大很多。

我對這種跨領域應用一直很感興趣。語言模型的本質如果是 pattern recognition，那化學結構也是一種 pattern。只是以前沒人真的把它接上實驗儀器而已。

來源：https://www.anthropic.com/research/making-claude-a-chemist

## 素材 3：刻意削弱自家 AI agent 的開發者

HN 上一篇 22 點的文章：一個開發者說他「故意 nerf 了自己的 coding agent」。

他說不是 agent 不夠強，是太強了。強到他開始不理解自己 codebase 裡的某些部分。與其讓 agent 一路狂寫，他選擇限制 agent 的權限，強迫自己停在「每個變更都看得懂」的範圍內。

這其實是一個很少被討論的議題：當 agent 的產出超越你的理解能力時，你還算是在「開發」嗎？還是只是在「部署」？

來源：HN Show HN (2026-06-05, 22pts)

## 素材 4：Hermes Agent — 開源 AI Agent 平台

Hermes Agent 被放上 HN，定位是「有持久記憶的開源 AI Agent」。支援多平台（Telegram、Discord、WhatsApp 等），可以自架、自訂 skills、跑 cron 任務。

跟 Claude Code 那種單機 CLI agent 不同，Hermes 更像是一個 agent OS：有 scheduler、有 plugin 系統、有 gateway 可以接各種通訊平台。

開源社群現在很需要這類基礎設施。不是每家公司都想把 agent 綁在單一模型供應商身上。

來源：https://hermes-agent.org/

## 素材 5：AI Agent 安全的多層政策框架

Tigera 發了一篇關於如何用多層政策來保護 AI agent 的文章。

核心概念：agent 不是單一 process，而是一個會呼叫 API、讀取檔案、執行 shell 指令的複雜系統。傳統的「防火牆擋外面」已經不夠用。他們提出分層：網路層、process 層、API 層、資料層，每個層都要有獨立的 policy。

隨著 agent 開始碰到 production 環境，安全問題會比功能問題更早爆。這篇值得追。

來源：https://www.tigera.io/blog/multi-layer-policy-for-securing-ai-agents/

## 素材 6：OpenClaw 公開安全改進

OpenClaw 發了一篇部落格文，講他們在安全性上做了哪些改進。包括 sandbox、permission model、audit log。

HN 上有人問「OpenClaw/Hermes 目前最大的問題是什麼」，剛好跟這篇呼應。社群最關心的幾個點：安全、穩定、跨平台支援。

開源 agent 生態現在百花齊放，但真正能進 production 的還不多。安全是最基本的門檻。

來源：https://openclaw.ai/blog/openclaw-security-in-public
